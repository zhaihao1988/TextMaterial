# 反馈训练脚本 - 设计文档

本文档为「基于用户反馈优化签文匹配」的机器学习脚本提供设计依据，供后续实现时参考。与 `匹配规则.md` 第 7 节「用户反馈与学习机制」配套使用。

---

## 1. 目标与产出

### 1.1 核心目标

- **回归打分**：根据历史反馈，训练模型预测「某条签文在某个 `req_key` 下被抽中后，用户可能给出的 `hit_score`（命中程度）」。
- **参数建议**：生成对 `match_weights` 与 `blind_safe` 的更新建议，供人工审阅后写回 `tanjing.json`。

### 1.2 产出物

| 产出 | 说明 |
|------|------|
| 训练好的回归模型 | 可加载用于预测候选签文的预期得分 |
| 统计报告 | 各签文/路由组合的平均 `hit_score`、出现次数、异常组合 |
| 参数更新建议 | JSON 格式，建议修改的 `match_weights` 与 `blind_safe`，需人工审核 |

---

## 2. 数据依赖

### 2.1 反馈日志：`mapping/feedback_log.jsonl`

- **格式**：JSON Lines，每行一条独立 JSON。
- **路径**：`{项目根}/mapping/feedback_log.jsonl`。
- **单条结构**（与 `匹配规则.md` 7.2 一致）：

```json
{
  "feedback_id": 1,
  "source": "liuzutanjing",
  "index": 37,
  "req_key": "樊笼-疲于奔命_Option_B",
  "timestamp": "2026-02-28T12:34:56",
  "hit_score": 5,
  "analysis_quality": 4,
  "blind_safe_suggestion": "keep",
  "comment": "命中感很强，但略偏锋利。"
}
```

| 字段 | 类型 | 用途 |
|------|------|------|
| `feedback_id` | int | 唯一标识 |
| `source` | str | 签文来源，与 tanjing 中 `source` 对应 |
| `index` | int | 签文编号，`source + index` 为全局主键 |
| `req_key` | str | 36 维路由键之一 |
| `timestamp` | str | ISO8601 时间戳 |
| `hit_score` | int | 1–5，命中程度（**主预测目标**） |
| `analysis_quality` | int | 1–5，解析质量（可选预测目标） |
| `blind_safe_suggestion` | str | `keep` / `should_be_blind_safe` / `should_not_be_blind_safe` |
| `comment` | str | 自由备注，可选 |

### 2.2 签文库：`tanjing.json`

- **格式**：JSON 数组，每项为一条签文。
- **路径**：`{项目根}/tanjing.json`。
- **与本脚本相关的字段**：

| 字段 | 用途 |
|------|------|
| `source` | 与 feedback 的 `source` 关联 |
| `index` | 与 feedback 的 `index` 关联 |
| `match_weights` | 字典，`{ req_key: number }`，用于特征 |
| `blind_safe` | 布尔，是否适合盲抽 |
| `ai_instruction` | 语义锚点，用于向量相似度（若启用语义特征） |

### 2.3 36 维路由键

- 标准 Key 列表见 `系统提示词.md`。
- 格式：`"<子场景>_Option_X"`，例如 `"樊笼-疲于奔命_Option_B"`。
- 共 12 子场景 × 3 Option = 36 个。

## 3. 当前数据局限与扩展建议

### 3.1 当前反馈中缺失的字段

- **`user_text`**：用户输入的描述文本（如「被裁员，感觉很委屈又很迷茫」）。
- 若有 `user_text`，可计算 `semantic_score = cosine_similarity(embed(user_text), embed(ai_instruction))`，作为强特征。
- **实现建议**：若未来支持「标签 + 文本」抽签流程，在保存反馈时增加可选字段 `user_text`，便于后续训练使用。

### 3.2 可立即使用的特征（无需扩展）

- `match_weights[req_key]`：当前规则权重。
- `req_key`：路由键（需编码）。
- `index` / `source`：签文标识。
- `blind_safe`：签文是否在盲抽池。
- 签文其他属性：`scene_category`、`dimension_v`、`dimension_e` 等（若需可编码加入）。

---

## 4. 特征工程

### 4.1 特征列表（按实现阶段）

#### 阶段一：仅用反馈 + 签文静态信息（无需 Embedding）

| 特征名 | 来源 | 类型 | 说明 |
|--------|------|------|------|
| `match_weight` | `tanjing[index].match_weights[req_key]` | float | 0 表示未配置 |
| `req_key_idx` | 36 维路由键的索引 | int | 0–35 |
| `req_key_onehot` | 36 维 One-Hot | vec | 可选，维度较高 |
| `index` | 签文 index | int | 需归一化或分桶 |
| `blind_safe` | 签文 `blind_safe` | 0/1 | |
| `scene_category` | 签文场景 | 类别编码 | 4 类 |
| `dimension_v` | V1–V4 | 类别编码 | |
| `dimension_e` | E1–E3 | 类别编码 | |

#### 阶段二：增加语义特征（需 Embedding）

| 特征名 | 来源 | 说明 |
|--------|------|------|
| `semantic_score` | `cosine_similarity(u_vec, v_vec)` | 需反馈中有 `user_text`，或仅在推理时使用 |

### 4.2 特征构建流程（伪代码）

```
for each line in feedback_log.jsonl:
    fb = parse(line)
    sutra = find_sutra(fb.source, fb.index)  # 从 tanjing 中查找
    if sutra is None:
        skip  # 签文可能已删除
    w = sutra.match_weights.get(fb.req_key, 0)
    features = {
        "match_weight": w,
        "req_key_idx": req_key_to_idx(fb.req_key),
        "index": fb.index,
        "blind_safe": 1 if sutra.blind_safe else 0,
        ...
    }
    label = fb.hit_score  # 或 (hit_score + analysis_quality) / 2
    yield (features, label)
```

### 4.3 编码约定

- **req_key**：维护 36 个标准 Key 的有序列表，`req_key_to_idx(key)` 返回 0–35。
- **类别字段**：使用 `sklearn.preprocessing.LabelEncoder` 或 One-Hot，训练时 fit 后保存编码器供推理使用。

## 5. 标签定义

### 5.1 主标签：`hit_score`

- 范围：1–5，连续数值。
- 含义：用户对「这条签文是否命中我的状态」的主观评分。

### 5.2 可选标签

- **`analysis_quality`**：1–5，解析质量。
- **综合分**：`(hit_score + analysis_quality) / 2`，若希望同时考虑两者。

### 5.3 分类任务（可选）

- `blind_safe_suggestion` 可单独做**分类模型**：预测 `keep` / `should_be_blind_safe` / `should_not_be_blind_safe`，用于生成 `blind_safe` 更新建议。

## 6. 模型选择（分阶段）

### 6.1 数据量建议

| 反馈条数 | 建议 |
|----------|------|
| < 200 | 仅做统计聚合，不做模型训练 |
| 200–500 | 线性回归 / 随机森林，严格防过拟合 |
| 500–2000 | 随机森林 / 轻量 MLP |
| > 2000 | 可尝试 2 层 MLP 或更复杂模型 |

### 6.2 模型选项

| 模型 | 库 | 适用场景 | 备注 |
|------|-----|----------|------|
| 线性回归 | sklearn | 基线、可解释 | 快速验证特征有效性 |
| 随机森林回归 | sklearn | 中等数据、非线性 | 可输出特征重要性 |
| 梯度提升 | XGBoost / LightGBM | 表格数据、效果较好 | 需安装额外依赖 |
| 小型 MLP | PyTorch / Keras | 数据较多时 | 2 层隐藏层即可 |

### 6.3 评估指标

- **MSE**（均方误差）：主指标。
- **MAE**（平均绝对误差）：直观理解「平均差几分」。
- **与规则分的相关性**：若模型分与 `match_weight` 高度相关，说明规则已较好，模型增益有限。

## 7. 训练流程

### 7.1 整体流程

```
1. 加载 feedback_log.jsonl、tanjing.json
2. 特征工程：构建 (X, y) 数据集
3. 划分训练集 / 验证集（如 8:2，按时间或随机）
4. 训练模型
5. 在验证集上评估
6. 保存模型 + 编码器
7. （可选）生成统计报告与参数更新建议
```

### 7.2 划分策略

- **随机划分**：`sklearn.model_selection.train_test_split`，`shuffle=True`。
- **按时间划分**：若希望评估「对未来反馈的预测能力」，可按 `timestamp` 排序，前 80% 训练、后 20% 验证。

### 7.3 过拟合防护

- 数据少时：强正则、早停、减少模型复杂度。
- 随机森林：限制 `max_depth`、`min_samples_leaf`。
- 神经网络：Dropout、L2 正则、Early Stopping。

## 8. 参数更新建议生成

### 8.1 基于统计的简单规则（无需模型）

- 对每个 `(index, req_key)` 组合，统计：平均 `hit_score`、出现次数。
- 若某组合出现次数 ≥ N 且平均 `hit_score` < 阈值，建议：降低 `match_weights[req_key]` 或移除该挂载。
- 若某签文被多次标记 `should_not_be_blind_safe`，建议：`blind_safe = false`。

### 8.2 基于模型的建议

- 对 tanjing 中每条签文、每个已配置的 `req_key`，用模型预测 `hit_score`。
- 若预测分持续偏低，可生成「建议下调权重」的条目。
- 需设定阈值与人工审核规则，避免自动写回。

### 8.3 建议输出格式（示例）

```json
{
  "match_weights_suggestions": [
    {
      "source": "liuzutanjing",
      "index": 37,
      "req_key": "樊笼-疲于奔命_Option_B",
      "current_weight": 60,
      "suggested_weight": 40,
      "reason": "该组合平均 hit_score=2.1，出现 15 次"
    }
  ],
  "blind_safe_suggestions": [
    {
      "source": "liuzutanjing",
      "index": 42,
      "current": true,
      "suggested": false,
      "reason": "5 次反馈中 4 次标记为 should_not_be_blind_safe"
    }
  ]
}
```

## 9. 与匹配逻辑的集成

### 9.1 当前规则（不变）

- 纯盲抽：`Final_Score = match_weights[req_key]`
- 有文本：`Final_Score = match_weights[req_key] + semantic_score * 1000`

### 9.2 模型介入方式（可选）

| 方式 | 说明 |
|------|------|
| 完全替代 | 用模型预测分替代 `Final_Score`，需充分验证 |
| 混合 | `Final_Score = α * 规则分 + (1-α) * 模型分`，α 可配置 |
| 辅助排序 | 规则筛选候选后，用模型分做精排 |

### 9.3 推理时特征

- 候选签文 + 当前 `req_key` +（若有）`user_text`。
- 构建与训练时相同的特征向量，调用模型 `predict`。
