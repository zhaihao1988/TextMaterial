# 训练集自动生成方案

本文档描述基于**本地大模型（LLM）**的盲抽场景模拟与自动化反馈流程，用于生成可迭代匹配策略与模型优化的**训练集**。流程依赖《匹配规则.md》中的盲抽逻辑，并与 `blind_draw_gui.py` 的抽签能力对齐。

---

## 1. 目标与背景

- **目标**：在无人值守或半自动条件下，批量生成「场景描述 → 标签选择 → 盲抽结果 → 质量反馈」的完整记录，形成训练集。
- **规模**：场景库**至少 200 条**形色各异的自然语言场景；每条场景抽 20 次签并打分，共 **200 × 20 = 4000 条**训练记录（若部分抽签无候选则略少）。
- **抽样要求**：**同一场景、同一 req_key 下，执行 20 次盲抽**；每次抽得 1 条签文后均由 LLM 对该次结果打分反馈，即每个场景产出 20 条训练记录（若某次无候选签文则该次记为「无结果」）。
- **用途**：训练集可用于后续分析匹配质量、调整 `match_weights` / `blind_safe`，或训练/微调「场景→标签」与「匹配质量」相关模型。
- **角色分工**：
  - **LLM**：生成自然语言场景、为场景选定路由键、对签文与场景的符合度进行分析并打分。
  - **程序**：按 `req_key` 调用现有盲抽逻辑（与 `blind_draw_gui.py` 一致），产出签文内容，并整理为结构化记录。
- **标签模糊性**：同一场景往往对应**多种都合理的标签**（例如同一子场景下选 Option_A 或 Option_C 皆可），因为 36 维标签设计本身偏模糊、归因（贪/嗔/痴）可多解。方案中需承认这一点，并在「选标签」与「抽签」环节给出明确处理方式（见 3.2）。

---

## 2. 整体流程概览

```
                                    │

└─────────────────────────────────────────────────────────────────────────┘
```

## 3. 步骤详述

### 3.1 Step 1：LLM 生成场景库

**目的**：得到一批**自然语言**的「用户处境」描述，作为后续选标签与打分的主体；规模足够大、多样性足够高，以支撑约 4000 条训练集。

**数量与多样性要求**：

- **至少 200 条**场景，条条**形色各异**，避免重复或高度雷同。
- **尽可能全面**地模拟真实用户群体：
  - **身份 / 职业**：学生、打工人、自由职业、创业者、体制内、退休、全职家长、蓝领、白领、服务业、创意行业等。
  - **性别与年龄**：男女、不同年龄段（青年、中年、中老年等）的典型处境与表达方式。
  - **情境类型**：职场压力、家庭关系、金钱得失、自我怀疑、人际冲突、边界问题、选择焦虑、倦怠、委屈、迷茫等，覆盖 4 大场景（樊笼 / 沉浮 / 尘缘 / 方寸）下的多种子场景与 36 维 req_key。
- 每条场景是一段**具体、可感**的叙述（第一人称或可代入的处境描述），避免空洞标签式句子。
- 示例（尘缘-力不从心）：

  > 我今天本来想在家里工作，但是父母要来看小孩，又要订酒店，安排中午吃饭，安排父母跟老婆父母见面等等一系列麻烦事，自己的事情干不了。

**输出格式**：

- 建议为 JSONL，每行一条：`scene_id`、`scene_text`，以及 **`target_category`**（本轮约定的大类简称，如 `职场/生存(樊笼)`），便于跑批时带出到每条反馈记录形成完整故事闭环。例如：

```json
{"scene_id": 1, "scene_text": "我今天本来想在家里工作，但是父母要来看小孩……", "target_category": "家庭/关系(尘缘)"}
```

- 也可先输出纯文本列表，由脚本再赋予 `scene_id`；使用 `--gen-scenarios` 时脚本会自动写入 `target_category`。

---

### 3.2 Step 2：根据场景选 req_key 并执行盲抽

**前提（标签模糊性）**：一个场景可能对应**多个都合理的 req_key**。例如「父母来看小孩、订酒店、安排两家见面、自己的事干不了」这类处境，既可能被归为「缘缚手脚，茧困深恩」选 **Option_C**，也可能被归为另一种归因选 **Option_A**；标签设计相对模糊，这是正常现象。因此 LLM 应允许输出多个合理选项，由程序按约定策略选用。

**输入**：

- 单条场景的 `scene_id`、`scene_text`。
- （可选）《系统提示词.md》中 36 维路由键与 Option 文案，供 LLM 参考。

**LLM 职责**：

- 根据 `scene_text` 判断该处境属于哪个**大场景 + 子场景**，以及**一个或多个**合理的**归因选项（Option_A/B/C）**。
- 输出**至少一个、可多个**标准路由键 `req_key`，格式：`"<子场景>_Option_X"`。例如：
  - 仅一个：`"尘缘-力不从心_Option_C"`（缘缚手脚，茧困深恩）
  - 多个合理：`["尘缘-力不从心_Option_A", "尘缘-力不从心_Option_C"]`（表示选 A 或选 C 都说得通）
- 若输出多个，建议标注主选与备选（或由程序视为等价，见下）。

**程序职责**：

- **多 req_key 的处理（策略 A）**：若 LLM 输出多个合理 `req_key`，程序**随机选其中一个**作为本场景当次的唯一 req_key，再对该 req_key 执行 20 次盲抽；同一场景多次运行或不同批次时可得到不同 req_key，训练集自然覆盖「同一场景可对应 A 或 C」。
- 对**选定的该 req_key**，执行 20 次盲抽（每次独立调用双重轮盘算法）：
  - 每次调用与 `blind_draw_gui.py` 一致的盲抽逻辑（见《匹配规则》4.1 及《双重轮盘算法设计.md》）：
    - 从 `tanjing.json` 中筛选 `blind_safe == true` 且 `match_weights[req_key] > 0` 的签文；
    - 先按权重分入 SSR/SR/R/N 四个分数档位，再在非空档位之间按预设出场权重（75/15/8/2）做第一重轮盘，随后在选定档位内等概率随机抽取 **1 条**签文（如该档位为空则退化为在所有非空档位的全集上等概率抽取）。
  - 若某次候选为空，该次记为「无可用签文」，可记入训练集但标记为「无结果」。
  - 每次抽中的签文均格式化为「回复内容」供 Step 3 使用，至少包含：`index`、`sutra_title`、`sutra_text`、`ui_translation`、`ui_mapping`、`ui_action`，以及签文中的 `source`（若有）。

**输出**（每场景共 20 次，每次一套）：

- `req_key`；
- 当次抽中的签文完整展示文本（或结构化字段）；
- 签文 `source`、`index`（用于反馈记录定位）；
- 建议增加 `draw_round`（1–20）标识当次是该场景下的第几次抽签，便于按场景聚合分析。

---

### 3.3 Step 3：LLM 对签文与场景符合度打分并生成反馈

**输入**：

- 同一条的 `scene_text`；
- 该次盲抽得到的签文内容：标题、原文、翻译、`ui_mapping`、微行动等。

**LLM 职责**：

- **分析**：签文内容（尤其是 `ui_mapping`、微行动）是否**语义上、情境上**贴合该场景；是否「说到点子上」、有无错配或过激。
- **打分与建议**（与《匹配规则》7.2 对齐）：
  - **命中程度 `hit_score`**：1–5，数值越大表示越「准」。
  - **解析质量 `analysis_quality`**：对 `ui_mapping` 等解析文本质量的评分，1–5。
  - **盲抽适合性 `blind_safe_suggestion`**：布尔值
    - `true`：适合盲抽（应纳入或保持盲抽池）；
    - `false`：不宜盲抽（建议移出盲抽池）。
  - **comment**（可选）：自由文本，记录边界情况或特别印象。

**程序职责**：

- 将 LLM 输出与**定位信息**合并为一条反馈/训练记录；
- 定位信息包括：`feedback_id`、`source`、`index`、`req_key`、`timestamp`，以及本方案扩展的 `scene_id`、`scene_text`、`draw_round`（1–20，表示该场景下第几次抽签）等。
- 每个场景共写入 **20 条**记录（对应 20 次抽签；若某次无结果则该条仅含场景与无结果标记）。

## 4. 训练集 / 输出数据格式

单条训练记录必须是一个**完整的故事闭环**，便于复盘、审查多样性并沉淀为黄金测试集；同时作为**反馈日志**与**训练样本**使用。

### 4.0 为什么必须存「完整故事闭环」？

跑批时若只记录打分（如 `hit_score: 1, comment: "签文未切中背锅的痛点"`），第二天看日志时既不知道当时生成了什么场景、抽到了哪条签，也无法判断该分数是否合理。因此写入 `feedback_log.jsonl` 的每一行必须包含：

- **target_category**：本轮约定的大类（如「职场/生存(樊笼）」），用于审查 Step 1 是否按类别生成；
- **scene_text**：大模型编的完整场景叙述（**无价之宝**，复盘与黄金集原料）；
- **selected_req_key**：模型为该场景选的标签，用于复盘 Step 2 分类是否准确；
- **drawn_item**：当次抽中的签（index + title），用于对照打分是否合理；
- **eval_feedback**：打分与备注，与上述上下文一起形成闭环。

**复盘的三大作用**：

1. **复盘 Step 2（选标签）准确率**：对比 `scene_text` 与 `selected_req_key`，一眼发现分类错误（如场景是「伴侣冷暴力」却选了樊笼）。
2. **审查 Step 1（生成场景）多样性**：把多轮 `scene_text` 拎出来看，检查是否偷懒（如 80 个都是项目经理/程序员）。
3. **沉淀黄金测试集**：高质量「场景 + 正确标签」+ 可信评分，可提取为 `{"场景": scene_text, "正确标签": selected_req_key}`，用于测试小模型或微调。

### 4.1 字段说明

**完整故事闭环（推荐优先使用）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | str | ISO8601 |
| `target_category` | str | 本轮约定的大类简称，如 `职场/生存(樊笼)` |
| `scene_text` | str | 自然语言场景描述（大模型编的故事） |
| `selected_req_key` | str | 模型为该场景选中的路由键 |
| `drawn_item` | object | 当次抽中的签：`{ "index", "title", "angle" }`，其中 `angle` 为视角编号，直接来自 `tanjing.json` |
| `eval_feedback` | object | 打分与建议：`{ "hit_score", "analysis_quality", "blind_safe_suggestion", "comment" }` |

**定位与扁平字段（与《匹配规则》7.2 一致，便于既有脚本）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `feedback_id` | int | 自增，唯一 |
| `source` | str | 签文来源，如 `liuzutanjing` |
| `index` | 签文 index | 与 `tanjing.json` 对应 |
| `angle` | int/str | 视角编号，与 `tanjing.json` 中的 `angle` 对应，用于区分同一题材下不同视角版本 |
| `req_key` | str | 本次使用的路由键（与 `selected_req_key` 同值） |
| `scene_id` | int/str | 场景库中的场景编号 |
| `draw_round` | int | 该场景下第几次抽签，取值 1–20 |
| `sutra_title` | str | 签文标题 |
| `ui_mapping` | str | 解析展示文本 |
| `ui_action` | str | 微行动 |
| `hit_score` | int | 1–5，命中程度 |
| `analysis_quality` | int | 1–5，解析质量 |
| `blind_safe_suggestion` | bool | true=适合盲抽，false=不宜盲抽 |
| `comment` | str | 可选备注 |

### 4.3 输出位置与格式

- **主输出**：跑批脚本将上述记录**追加**到 `mapping/feedback_log.jsonl`，每条必须包含完整故事闭环（`target_category`、`scene_text`、`selected_req_key`、`drawn_item`、`eval_feedback`）。
- 场景库（如 `scenarios.jsonl`）生成时建议一并写入 `target_category`，以便跑批时带出到每条反馈记录中。
- 若需与人工反馈区分，可通过 `scene_id` / `draw_round` 等扩展字段或单独训练集文件（如 `mapping/training_set.jsonl`）做分流；程序读取时通过字段存在与否区分「自动生成」与「人工反馈」。

## 5. 与现有组件的对应关系

| 组件 | 关系 |
|------|------|
| 《匹配规则.md》 | 盲抽逻辑严格按 4.1（无文本、仅 req_key、双重轮盘 + 分层卡池，每次返回 1 条）；反馈字段与 7.2 对齐。 |

## 6. 实现要点（后续开发时参考）

1. **场景生成**：为 LLM 设计 prompt 时明确要求**至少 200 条**、**形色各异**：按身份/职业/性别/年龄与 4 大场景均衡分布，避免重复与雷同；可分批生成（如每批 20–50 条、多轮生成并去重），最终合并为 JSONL，保证总量 ≥ 200。
2. **选标签**：将《系统提示词.md》中的 12 子场景 + 36 Option 摘要放入 prompt，让 LLM 从场景文本映射到**一个或多个**合理的 `req_key`（因标签模糊性，同一场景选 A 或 C 等均可）；输出需校验为合法 36 维之一，多个时按**策略 A**随机选一个作为当次 req_key，再对该 req_key 执行 20 次盲抽。
3. **盲抽**：直接复用 `blind_draw_gui.py` 中的 `load_tanjing_items()`、`blind_draw_once()`；无 GUI 时可单独脚本 import 或通过子进程调用。
4. **打分**：为 LLM 提供「场景 + 签文展示内容」和明确的 1–5 分定义、三选一盲抽建议，输出结构化 JSON 便于解析写入 JSONL。
5. **幂等与追溯**：建议每条训练记录保留 `scene_id`、`req_key`、`draw_round`、`timestamp`，便于去重与按场景/签文聚合分析。
6. **20 次/场景**：对每个场景在选定 `req_key` 后循环 20 次盲抽，每次抽签后立即调用 LLM 打分并写入一条记录；实现时注意 `feedback_id` 自增与 `draw_round` 的递增。
7. **报错与重跑**：跑批时可能出现 timeout 等错误。实现上应保证：**一个场景没搞定就下一个**，选标签或打分失败时跳过当前场景/当次抽签并继续，无需彻底重头来；每条仅在构造完整 record 后写入 feedback_log 一行；若因进程中断等出现不完整的一行，删掉该行即可。单次 LLM 调用可自动重试 1 次。`--overwrite-scenarios` 仅用于需要清空并重新生成场景文件时，非报错后的必选。

## 7. 后续扩展（可选）

- 利用「同一场景 20 次」的样本，统计同一 `req_key` 下签文分布、命中分分布，用于估计匹配稳定性与轮盘赌权重合理性。
- 将训练集用于统计与规则迭代：如《匹配规则》7.4 所述，分析 `hit_score`、`blind_safe_suggestion`，生成对 `match_weights` / `blind_safe` 的更新建议，经人工审核后写回 `tanjing.json`。
- 用「场景 + req_key」或「场景 + 签文 + 反馈」微调/训练模型，用于场景→标签预测或匹配质量预估。
