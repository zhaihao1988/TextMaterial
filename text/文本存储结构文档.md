## 「抽个签」文本存储结构文档（基于 tanjing.json v1）

### 1. 总体结构说明

当前 `tanjing.json` 顶层是一个数组，每个元素代表一条**签文节点**（可以视作一个 Node）。  
在现阶段，每条记录是**平行的**，尚未显式区分 Root / Shard，但字段已经天然分成几大域：

- **场景与情绪标签域**：`tag`, `scene_category`, `emotion_state`, `dimension_v`, `dimension_e`
- **内容/展示域（Content & UI）**：`sutra_title`, `sutra_text`, `ui_translation`, `ui_mapping`, `ui_action`
- **盲抽路由域（Routing Meta）**：`blind_safe`, `match_weights`
- **语义检索域（Vector Meta）**：`ai_instruction`
- **元数据域（Meta）**：`source`, `index`

后端三大核心流程与这些字段的对应关系：

- **纯盲抽（无文本）**：只依赖 `blind_safe` + `match_weights[req_key]`。
- **标签 + 文本精排**：在标签粗筛基础上，用 `ai_instruction` 做 Embedding 相似度排序。
- **内容展示 / 编辑工具**：完全依赖 `sutra_*` 与 `ui_*` 系列字段。

---

### 2. 字段字典与含义（按域分组）

#### 2.1 场景与情绪标签域

- **`tag`**：`String`  
  - 含义：12 个基础标签之一，形如 `"方寸-妄念纷飞"`、`"樊笼-形如槁木"`。  
  - 作用：表示该签文的**原生主场景**；也是用户前端第二步选择的子场景之一。  
  - 关系：
    - 与 `scene_category` 一一对应（4 大场景的大类归属）。  
    - 与 `match_weights` 中的 Key 空间属于同一族（Key = 子场景 + Option）。

- **`scene_category`**：`String`  
  - 示例：`"自我/内省"`, `"职场/生存"`, `"家庭/亲密关系"`, `"外物/得失"`。  
  - 作用：4 大场景维度，用于粗粒度统计、预过滤，以及与产品文案对齐。

- **`emotion_state`**：`String`  
  - 含义：从用户主观体验角度，对当前签文对应的“心理机制”做一段自然语言描述。  
  - 使用场景：
    - 编辑/策划用于快速回忆该签文对应的心理状态；  
    - 将来可用于 LLM 辅助标注/聚类。

- **`dimension_v`**：`String`（离散枚举）  
  - 示例：`"V1 对抗态"`, `"V2 悬浮态"`, `"V3 沉坠态"`, `"V4 舒展态"`。  
  - 作用：纵向“情绪姿态”维度，帮助构建心智光谱。

- **`dimension_e`**：`String`（离散枚举）  
  - 示例：`"E1 高能"`, `"E2 中能"`, `"E3 低能"`。  
  - 作用：能量水平维度，用于后续更细致的推荐/统计。

---

#### 2.2 内容/展示域（Content & UI）

这些字段只用于前端显示与人工阅读，不直接参与路由或权重计算（向量模型只读 `ai_instruction`）。

- **`sutra_title`**：`String`  
  - 含义：签文标题，4–8 字左右，如 `"菩提本净"`, `"艰辛卖柴"`。  
  - 作用：前端主标题、工具中快速识别。

- **`sutra_text`**：`String`  
  - 含义：原典原文（六祖坛经中的一句或数句）。  
  - 作用：作为“根文本”呈现给用户。  
  - 备注：未来如果 Root / Shard 共用同一段原文，可以允许多个节点重复该字段。

- **`ui_translation`**：`String`  
  - 含义：白话直译，对 `sutra_text` 的语义还原。  
  - 作用：降低理解门槛，属于“基础解释层”。

- **`ui_mapping`**：`String`  
  - 含义：围绕当前签文所针对的**现代心理场景**做的长篇解析。  
  - 特点：
    - 明确站在用户当前的情境里，说“你此刻……”；  
    - 不参与任何路由/匹配计算，只作为展示 & 体验内容。

- **`ui_action`**：`String`  
  - 含义：物理级“微行动”指令。  
  - 作用：
    - 将抽象认知落地为具体可执行动作；  
    - 一般是今日即可执行的简短行为。

> 说明：后续如果要支持“同一签文多视角解释”，可以在新的 Node 设计上增加标识（例如 `node_type` 或 `interpretation_type`），并在不同节点中写入不同的 `ui_mapping` / `ui_action`，而不需要改变这几个字段本身的含义。

---

#### 2.3 盲抽路由域（Routing Meta）

- **`blind_safe`**：`Boolean`（当前数据中为 `"True"`/`"False"` 字符串，后续可迁移为真实布尔）  
  - 含义：该签文是否适合进入**盲抽池**。  
  - 使用：
    - 盲抽时必须满足 `blind_safe == true` 才会进入候选集合；  
    - 即使 `match_weights[req_key] > 0`，只要 `blind_safe == false`，就不会在盲抽中出现。  
  - 实际意义：
    - 允许同一类内容中存在**极度尖锐/冷酷**的解析，而只把温和/通用版本放入盲抽。

- **`match_weights`**：`Object<String, Number>`  
  - 结构：以 36 维路由键为 Key 的字典：
    - Key 形如：`"方寸-妄念纷飞_Option_C"`、`"樊笼-疲于奔命_Option_A"`；  
    - Value 为该签文在此路由下的权重分数（建议 0–100）。  
  - 规则（与《匹配规则.md》一致）：
    - 每条签文最多配置约 **3–6 个 Key**，避免概率被过多跨界挂载稀释；  
    - 必须有且仅有一个 Key 权重为 **100**（原生主场景“绝杀签”）；  
    - 其他跨界权重通常在 **30–70** 之间，用于“跨界点醒”。  
  - 盲抽使用（结合当前双重轮盘算法）：
    - 根据 `match_weights[req_key]` 得到该签文在此路由下的分数 `w`；  
    - 按 `w` 分档到 `SSR (w >= 100)` / `SR (80 <= w < 100)` / `R (60 <= w < 80)` / `N (0 < w < 60)` 四个桶；  
    - 再经过双重轮盘（先抽档位，再在档位内等概率抽一条）。

---

#### 2.4 语义检索域（Vector Meta）

- **`ai_instruction`**：`String`  
  - 含义：供 Embedding 模型使用的“语义锚点”文本。  
  - 特点：
    - 包含大量“情绪黑话 + 大白话标签”，例如：
      - `用户遭遇[失眠、内耗、选择困难...]，产生[极度焦虑...]，本质是[...]，系统将引导其[...]`  
  - 用途：
    - 向量化后作为签文节点在向量库中的代表；  
    - 在“用户输入长文本诉苦”的场景下，计算相似度并做精排。

> 后续如果要加入 `scene_filter` / `emotion_category` 等向量预过滤字段，可以在这一域中新增，不影响现有字段。

---

#### 2.5 元数据域（Meta）

- **`source`**：`String`  
  - 示例：`"liuzutanjing"`。  
  - 作用：
    - 标记来源语料（如“六祖坛经”、“金刚经”等）；  
    - 便于在多来源合并时做区分统计。

- **`index`**：`Number`  
  - 含义：整数编号，用于在当前语料集合内标识一段**原文 + 翻译**。  
  - 约定：
    - 同一段原文的所有不同角度，`index` 相同；  
    - 即：`index` 标识的是“经文片段”，不是“解释角度”。

- **`angle`**：`Number`  
  - 含义：同一段原文下的**视角编号（Angle）**，从 1 开始递增，例如 1、2、3、4…。  
  - 作用：
    - 在物理存储上，所有角度仍然是**平行节点**；  
    - 区分“通用解释 vs 不同场景下的特化解释”等；  
    - 对于只有一个解释版本的经文，可以统一设为 `1`。  
  - 与盲抽的关系：
    - `angle` 与 `blind_safe` **没有强制绑定关系**；  
    - 某些经文本身就极为锋利定向时，即便是 `angle: 1` 也可以设为 `blind_safe: false`；  
    - 也允许多个不同 `angle` 都是 `blind_safe: true`，或该原文下所有角度都不适合盲抽。
  - 唯一性约定：
    - 一个节点的全局唯一标识可以取为：`(source, index, angle)`；  
    - 其中 `(source, index)` 决定“是哪一段原文”，`angle` 决定“从哪个角度解读”。

### 3. 字段之间的关键关系

1. **`tag` 与 `match_weights` 的关系**  
   - `tag` 告诉你这条签文**原生最典型的场景**；  
   - `match_weights` 允许这条签文挂载到**多个路由键**上，用不同的权重参与分发；  
   - 一般来说：
     - `tag + Option_X` 对应的那个 Key，会被赋值为 100；  
     - 其他场景/Option 的 Key，根据“跨界合理性”给 30–80 不等。

2. **`scene_category` 与 4 大场景的关系**  
   - `scene_category` 只控制“属于哪一大类”（职场/生存、外物/得失、家庭/亲密关系、自我/内省）；  
   - `tag` 是更细的子场景；  
   - 这两者配合起来，给向量域和统计分析提供“分类标签”。

3. **盲抽流程中的字段链路**  
   - 输入：前端三层选择 → `req_key`（如 `"方寸-妄念纷飞_Option_C"`）；  
   - 过滤：  
     - 仅保留 `blind_safe == true` 且 `match_weights[req_key] > 0` 的签文；  
   - 分档：  
     - 用 `match_weights[req_key]` 的数值 `w` 决定其落入 SSR/SR/R/N 桶；  
    - 双重轮盘：  
      - 第一轮：用固定档位权重 `SSR(75)/SR(15)/R(8)/N(2)` 在“非空档位”间抽池子；  
      - 第二轮：在池子内用等概率抽一条签文；  
   - 输出：展示 `sutra_title` / `sutra_text` / `ui_translation` / `ui_mapping` / `ui_action` / `source` / `index` / `angle`。

4. **语义检索流程中的字段链路**  
   - 输入：用户长文本 `user_text`；  
   - Embedding：  
     - 对 `user_text` → `u_vec`；  
     - 对每条签文的 `ai_instruction` → `v_vec`；  
   - 相似度：`semantic_score = cosine_similarity(u_vec, v_vec)`；  
   - （如结合标签粗排）可选融合公式：  
     - `Final_Score = match_weights[req_key] + semantic_score * 1000`；  
   - 排序：按最终分数排序，Top1 或 TopN 作为结果。

5. **未来 Node 扩展与当前字段的兼容**  
   - 将来如果要落地 Root / Shard / 多视角解释，可以新增：
     - 如 `node_id`, `node_type`, `parent_id`, `progression_level` 等结构字段；  
   - 现有的这些字段（尤其是 `sutra_*` / `ui_*` / `match_weights` / `ai_instruction`）都可以**原封不动复用**；  
   - 只是由“当前每条记录 = 一个 Node”进一步升级为“Node + 显式关系”的形式。
